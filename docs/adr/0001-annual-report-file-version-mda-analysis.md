# Redefine Annual Report Analysis Around File Versions and Management Discussion

We will organize the product around **AnnualReport** as the primary business object, with one or more **FileVersions** as uploaded PDF sources and **AnalysisRuns** as executions that produce the current **AnalysisResult** for a specific FileVersion. Upload now performs strict admission for a complete Chinese annual report identified by normalized stock code, report year, and company full name, instead of deferring identity binding until analysis starts.

AnalysisResult will no longer be a full-report bundle of overview, hard rules, financial analysis, extracts, and generated Markdown. It will analyze only the third section, **"管理层讨论与分析"**, by building a persisted structured evidence package of source sections, text spans, structured tables, figure assets, and evidence references; the validated structured outline becomes the source for interactive report rendering, QA, and Markdown/ZIP downloads. This keeps the user experience centered on the annual report while preserving verifiable evidence and avoiding the older file/task-centric and whole-PDF analysis model.

Prototype validation confirmed that FileVersion display state should be inferred from the current AnalysisResult first and the latest AnalysisRun second, rather than stored independently. Generated report assets, the EvidencePackage, and Chroma index state are lifecycle-owned resources of AnalysisRun/AnalysisResult, not separate user-visible business objects; this trades direct artifact selection for simpler consistency repair and cleanup when a run stops, fails, loses its result, or is deleted.

AnalysisResult persists one EvidencePackage as the canonical evidence registry for source sections, text spans, tables, figures, and section locator evidence. Report detail, structured_outline, Markdown/ZIP, and QA are projections over that registry and must not mint separate evidence ids or persist projected asset URLs; locator evidence stays separate from analysis and QA evidence because it may include table-of-contents or boundary-heading material outside the ManagementDiscussionAnalysisSection body.

Implementation note: the analysis execution flow should sit behind one backend Module, `ManagementDiscussionAnalysisExecution`, rather than being spread across `MdaAnalysisService`, section-location helpers, figure extraction, outline validation, and QA-index preparation. The execution Module should own one FileVersion/AnalysisRun analysis pass end to end: extract the ManagementDiscussionAnalysisSection, build the EvidencePackage, analyze figures, validate the outline, prepare QA index inputs, and return the AnalysisResult draft. `MdaAnalysisService` should remain the higher-level application facade that coordinates lifecycle transitions and persistence around that execution.

Implementation note: the backend should expose a single facade for current AnalysisResult access and asset reads so report detail, QA, download, table resource access, and figure image access all share the same file-version visibility and ownership checks. This keeps the seam shallow for callers while concentrating current-result lookup and asset validation in one place.

Implementation note: EvidencePackage projection should sit behind one backend Module, `EvidencePackageProjection`, rather than being split across the registry object, report downloads, QA retrieval, and current-result access. The registry owns source sections, text spans, tables, figures, and validation; the projection Module owns report detail shaping, QA index document shaping, Markdown/ZIP evidence rendering, and table/figure read models. This keeps canonical evidence storage deep and the derived views behind a small facade.

Implementation note: upload intake should sit behind one backend Module, `AnnualReportUploadIntake`, rather than being split across the route handler, admission service, persistence helper, and source-PDF write. The route should only forward the request and response; the intake Module should own admission, duplicate detection, AnnualReport/FileVersion creation, and source PDF write, while leaving PDF text extraction and storage adapters behind it.

Implementation note: the AnnualReport library user flows should sit behind one backend Module, `AnnualReportLibrary`, rather than being split across route handlers, list queries, presenters, and delete-confirmation shaping. The route should only forward the request and response; the library Module should own AnnualReport list projection, FileVersion delete confirmation shaping, AnnualReport delete confirmation shaping, delete response shaping, and the startup-visible library cleanup entrypoint, while leaving raw persistence queries and lifecycle cleanup rules behind it. The interface should be a facade for AnnualReport browsing and management, not a thin list helper.

Implementation note: the AnalysisResult detail interactions should sit behind one frontend Module, `AnalysisResultDetailInteraction`, rather than being split across the report detail panel, local table-loading state, QA session state, QA submit flow, and report-detail-local error handling. The panel should only render and forward events; the interaction Module should own table detail loading, QA session accumulation, QA submit flow, QA copy/download behavior, and local error state, while leaving report rendering and static labels as presentation helpers behind it. The interface should be a facade for the open report view, not a table-and-chat helper.

Implementation note: AnalysisRun lifecycle rules should sit behind one backend Module, `AnalysisRunLifecycle`, rather than being split across route handlers, FileVersion state helpers, analysis orchestration, and library deletion. Starting, stopping, retry eligibility, result-deleted transitions, run-owned artifact cleanup, FileVersion/AnnualReport deletion cleanup, and startup repair all depend on the same status and ownership invariants, so the lifecycle seam should concentrate those transitions while leaving PDF extraction, outline generation, QA indexing, and source-PDF deletion as adapters or callers behind it. The interface should be a facade over business actions, not a status-field helper.

Implementation note: the frontend should expose one facade for FileVersion user workflow, `FrontendFileVersionWorkflow`, rather than splitting action orchestration, API call sequencing, status refresh, confirmation flows, and current report cleanup across the page component. The page should only render and forward user events; the workflow Module should own the order of FileVersion actions, loading and refresh behavior, confirmation handling, and clearing the current report when a FileVersion or its analysis result is removed. Presentation helpers for labels and action metadata can stay separate.
